# 바로씻 테스트 에이전트 — 쉬운 사용설명서

> 천천히 읽으면서 이해하시라고 만든 문서입니다. 어려운 용어는 그때그때 풀어서 설명합니다.

---

## 1. 한마디로 뭐예요?

> **"바로씻 앱이 제대로 작동하는지 자동으로 시험 봐주는 도우미"** 입니다.

원래는 사람이 앱을 일일이 눌러보며 "이 기능 되나? 저 기능 되나?" 확인해야 했는데,
그걸 **대신 해주고 결과를 정리해주는 도구**를 만든 겁니다.

---

## 2. 시험에 비유하면 (가장 쉬운 설명)

학교 시험으로 비유하면 이렇습니다:

| 우리가 만든 것 | 시험으로 치면 | 하는 일 |
|---|---|---|
| `qa/checklist.json` | **시험 문제지** | "확인할 항목 63개" 목록 |
| `qa/playbook.md` | **시험 감독 지침서** | "어떻게 채점하는지" 방법 |
| `qa/runner.mjs` | **자동 채점기(OMR)** | 사람 없이 빠르게 일부 자동 채점 |
| `barosit-qa-tester` (에이전트) | **대리 시험 감독관** | 똑똑하게 전체를 시험 보고 결과 정리 |
| `qa/results/...json` | **성적표** | 항목별 합격/불합격 결과 |

즉, **문제지(체크리스트)**가 있고, **채점기(러너)**와 **감독관(에이전트)**이 그걸 보고 시험을 본 뒤 **성적표(결과)**를 줍니다.

---

## 3. 왜 이렇게 만들었나요? (중요)

전에는 **이 채팅창에서** 시험을 봤습니다. 그런데 채팅창은 대화가 길어지면
기억할 수 있는 양(context)에 한계가 있어서, 시험을 끝까지 못 보거나 앞 내용을 잊어버립니다.

그래서 **핵심 아이디어**는 이겁니다:

> 📌 **시험 내용을 "대화"가 아니라 "파일(qa 폴더)"에 적어둔다.**
> 그러면 매번 **새 기억(깨끗한 상태)**으로 시작한 도우미가 그 파일만 읽고 시험을 보면 되니,
> 대화가 아무리 길어져도 상관없습니다.

이게 핵심입니다. **문제지·지침·성적표가 전부 파일로 남아 있어서**, 채팅 기억에 의존하지 않습니다.

---

## 4. 도우미가 두 종류예요 (러너 vs 에이전트)

같은 시험을 보는 도우미가 **둘** 있습니다. 용도가 다릅니다.

### 🅰 에이전트 (`barosit-qa-tester`) — 똑똑한 감독관
- **AI가** 직접 앱 화면을 띄우고, 버튼을 눌러보고, 코드를 읽고 판단합니다.
- **거의 모든 항목**(약 50개)을 시험 볼 수 있습니다.
- 사람이 해야 하는 것(카메라 앞에서 자세 잡기 등)은 **"이것 좀 해주세요"** 하고 물어봅니다.
- 채팅에서 부릅니다.

### 🅱 러너 (`npm run qa`) — 빠른 자동 채점기
- **AI 없이** 정해진 검사만 빠르게 돌립니다 (현재 16개).
- 데이터베이스 관련 검사(가입·권한·게시판 등)를 **확정적으로** 즉시 확인.
- 화면을 눈으로 보는 검사는 못 합니다 → 그건 "에이전트한테 맡기세요"라고 표시만 합니다.
- 명령어 한 줄로 돌립니다.

> 👉 **빠른 점검은 러너(🅱), 꼼꼼한 전체 점검은 에이전트(🅰)** 라고 기억하세요.

---

## 5. 실제로 어떻게 쓰나요? (상황별)

> ⚠️ **먼저 — 어떤 Claude를 쓰는지에 따라 호출법이 다릅니다.**
> - **Claude Code (CLI/IDE)**: `@agent-barosit-qa-tester`, `/agents`, `/barosit-qa` 같은 기능이 됩니다.
> - **Claude 데스크톱 앱 / 일반 대화**: 위 기능들이 **없습니다**(`/agents` 안 나옴). 대신 → **`qa/PROMPT.md` 의 시작 프롬프트를 새 대화에 붙여넣으세요.** 효과는 똑같습니다.
> - **터미널만 있으면**: `npm run qa` — 환경 무관하게 항상 됩니다.

### 상황 1) 그냥 빠르게 핵심만 확인하고 싶다
터미널에 입력:
```
npm run qa
```
→ 16개 항목을 몇 초 만에 검사하고 합격/불합격을 보여줍니다. (AI·재시작 불필요)

### 상황 2) AI로 꼼꼼하게 전체를 시험 보고 싶다
**먼저 Claude Code를 한 번 재시작**하세요. (새로 만든 감독관을 등록하려고)
그다음 채팅창에 이렇게 **요청과 함께** 입력:
```
barosit-qa-tester 로 전체 자동검증 돌려줘
```
→ AI 감독관이 **별도 창(독립된 기억)**에서 시험을 보고, 이 대화창엔 **요약만** 돌려줍니다.

> ⚠️ `@barosit-qa-tester` 만 치고 엔터 = 안 됩니다. **"무엇을 하라"는 말**을 꼭 붙이세요.
> 예) `barosit-qa-tester 로 Billing 카테고리만 검증해줘`

### 상황 3) 사람이 직접 해야 하는 것까지 전부 검증하고 싶다
```
barosit-qa-tester 로 scope=full 전체 검증해줘
```
→ 자동 항목은 알아서 하고, **사람이 필요한 13개**에서 한 줄씩 부탁합니다.
예: *"메인 창 X 버튼 눌러주세요"*, *"카메라 켜고 거북목 자세 취해주세요"*, *"구글 로그인 동의까지 해주세요"*
→ 하시면 AI가 결과를 보고 합격/불합격을 매깁니다.

## 6. `auto` 와 `full` 차이 (쉽게)

이건 **"어디까지 시험 볼까?"** 를 고르는 것뿐입니다. 설정 파일 같은 거 없습니다.
**요청할 때 말로** 정합니다.

| 고르기 | 뜻 | 사람이 할 일 |
|---|---|---|
| `auto` (기본값) | 사람 손 안 가는 것만 (~50개) | **없음** (그냥 둬도 됨) |
| `full` | auto + 사람이 해야 하는 것 (13개 추가) | 카메라 자세·앱 클릭·로그인 등 **~13번 도와주기** |

- 아무 말 안 하면 `auto` 입니다.
- 전부 다 하고 싶으면 요청에 `scope=full` 이라고 붙이면 됩니다.

---

## 7. "사람이 꼭 해야 하는 것"은 뭔가요?

AI가 도저히 대신 못 하는, **물리적/외부 행동**들입니다 (총 13개):

- 📷 **카메라 앞에서 자세 잡기** — 거북목·고개 돌리기 (사람 몸이 필요)
- 🖥 **데스크톱 앱 네이티브 버튼** — 창 닫기/최소화, 트레이 메뉴 (실제 클릭이 필요)
- 🔑 **로그인 동의 / 실제 결제** — 구글·카카오 동의 화면, 토스 카드결제 (외부 서비스)

이건 `full` 로 검증할 때 AI가 **그 순간에만** "이것 좀 해주세요" 하고 부탁합니다.
나머지는 전부 자동입니다.

---

## 8. 결과(성적표)는 어디서 봐요?

시험을 보면 `qa/results/날짜시간.json` 파일이 생깁니다. 이걸 **앱 화면에서 예쁘게** 볼 수 있습니다:

1. 브라우저에서 QA 대시보드 열기 → 주소 끝에 `#/qa`
2. **"📥 결과 불러오기 (Import JSON)"** 버튼 클릭
3. 방금 생긴 결과 파일 선택
→ 항목마다 **합격(초록)/불합격(빨강)** 과 메모가 채워집니다.

> 메모 앞 태그 뜻: `[실증]`=실제로 돌려서 확인, `[코드]`=소스 확인, `[단위테스트]`=테스트 통과, `[수동필요]`=사람이 해야 함.

## 9. (나중 이야기) "Playwright" 는 뭐예요?

지금 **화면을 눈으로 보는 검사 26개**는 **AI 감독관(🅰)**만 할 수 있습니다.
(러너 🅱는 화면을 못 봐서 "AI한테 맡기세요"라고 넘깁니다.)

**Playwright** = 사람처럼 브라우저를 자동으로 조작하는 도구입니다.
이걸로 그 26개를 **미리 스크립트로 짜두면**, 러너(🅱)도 **AI 없이** 화면 검사를 확정적으로 할 수 있게 됩니다.

- 장점: `npm run qa` 만으로 더 많이(16 → 40여 개) 자동 검사 + 자동화(CI)에 유리
- 단점: 26개를 코드로 만들고 관리해야 함
- **지금은 안 해도 됩니다.** AI 감독관이 이미 해주니까요. **나중에 자동화를 본격화할 때** 고려할 선택사항입니다.

## 10. 용어 사전 (헷갈리면 여기)

- **에이전트(agent)**: AI 도우미 한 명. 여기선 `barosit-qa-tester` 라는 전용 시험 감독관.
- **러너(runner)**: AI 없이 정해진 검사를 돌리는 프로그램 (`npm run qa`).
- **체크리스트(checklist)**: 확인할 항목 목록 (시험 문제지).
- **scope(스코프)**: 시험 범위. `auto`(자동만) / `full`(사람 도움 포함).
- **context(컨텍스트)**: AI가 한 번에 기억할 수 있는 양. 대화가 길면 부족해짐 → 그래서 파일에 저장.
- **Supabase**: 앱의 데이터베이스/로그인 서버. 검사할 때 켜져 있어야 함 (`supabase status` 로 확인).
- **integration/runtime/unit/code/manual**: 각 항목을 "어떤 방법으로 검사하는지" 분류.
  - integration=DB로 실제 확인, runtime=화면 확인, unit=코드 자동테스트, code=소스 읽기, manual=사람 필요.

## 11. 자주 막히는 것 (FAQ)

**Q. `@barosit-qa-tester` 쳤는데 아무 일도 안 일어나요.**
→ (1) Claude Code를 **재시작**했는지 확인 (재시작해야 등록됨). (2) 이름 뒤에 **할 일**을 붙였는지 확인.

**Q. "Supabase 미가동" 이라고 나와요.**
→ 터미널에서 `supabase start` 후 다시 시도. (`supabase status` 로 켜졌는지 확인)

**Q. 결과가 63개가 아니라 적게 나와요.**
→ 화면 상단 플랫폼 필터가 "전체 플랫폼"인지 확인. 또는 옛 화면(barosit.com)이 아니라 **로컬(localhost)** 인지 확인.

**Q. 그냥 제일 간단하게 한 번 해보고 싶어요.**
→ 터미널에 `npm run qa` 입력. 이게 제일 쉽습니다.

## 12. 한 장 요약

```
빠른 점검   →  npm run qa
전체 점검   →  (재시작 후) "barosit-qa-tester 로 전체 자동검증 돌려줘"
완전 점검   →  (재시작 후) "barosit-qa-tester 로 scope=full 전체 검증해줘"
결과 보기   →  앱 #/qa → 결과 불러오기(Import) → qa/results 의 파일 선택
```

문제지(checklist)·지침(playbook)·성적표(results)가 **전부 파일**로 남으니,
대화가 길어져도 **언제든 새로 깨끗하게** 시험을 볼 수 있습니다. 그게 이 도구의 핵심입니다.
