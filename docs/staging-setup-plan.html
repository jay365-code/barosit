  • 보안성 확보: 어드민 권한 및 RLS 파괴적 검사의 실 운영 서버 유출 원천 차단
  • 데이터 오염 방지: 테스트 계정 생성/삭제 등의 찌꺼기 없는 깔끔한 운영 DB 보존
  • 연동 안정성: Toss Payments 모의(Sandbox) 결제 및 간편 로그인 공급자 연결 실증
인프라 격리 전략

기존 Cloudflare와 Supabase 계정을 연동하되, 격리 목적에 부합하게 설정을 다르게 조합합니다.

  • Supabase: 완전 격리된 별개의 신규 프로젝트(barosit-staging) 개설
  • Cloudflare: 기존 프로젝트 내 Preview 분기 배포 기능 및 도메인 분리 적용
  • Tauri Desktop: 스테이징 환경 변수가 바인딩된 테스터 전용 바이너리 빌드

인증(Auth) 환경 설정 요구사항

로그인 및 회원 정보가 다른 환경과 꼬이지 않도록 이중 보안 콜백 설정을 구축해야 합니다.

1. Google Cloud Console 및 Kakao Developers 설정

인증 공급자(Google, Kakao)는 사전 허용된 백엔드 콜백 URL로만 인증 토큰을 전달합니다.

  • 승인된 리디렉션 URI (Google / Kakao): 
    https://<staging-supabase-project-ref>.supabase.co/auth/v1/callback
    ※ 기존 운영용 OAuth 클라이언트에 위 주소를 추가로 등록하거나, 스테이징용 간편로그인 API 앱을 개발자 콘솔에 별도로 생성해 등록합니다.

2. Supabase Staging 대시보드 설정 (백엔드)

인증을 완수한 Supabase 백엔드가 사용자를 staging.barosit.com 프론트엔드로 안전하게 귀환시키는 화이트리스트 주소입니다.

  • 설정 위치: Supabase Console ➔ Authentication ➔ URL Configuration
  • 설정 세부사항:
    • Site URL: https://staging.barosit.com
    • Redirect URLs: https://staging.barosit.com/**

3. 이메일 가입 자동 인증 완화 설정 (테스트 편의 기능)

자동 QA 에이전트가 이메일 로그인/가입을 검증할 때 이메일 인증 절차(Magic Link 발송)를 우회하여 빠르게 검증하도록 설정합니다.

  • 설정 위치: Supabase Console ➔ Authentication ➔ Providers ➔ Email
  • 설정 세부사항: Confirm email 설정을 비활성화(Off) 처리하여 가입 즉시 active 계정이 되도록 구성합니다.
빌링(Billing - Toss Payments) 연동 설정 요구사항

실제 결제가 일어나지 않는 토스의 테스트용 **샌드박스(Sandbox)** API 키를 등록하고, 결제 결과가 올바르게 피드백되도록 웹훅(Webhook) 경로를 개설해야 합니다.

1. Toss Payments Sandbox API 키 분리

프론트엔드와 백엔드 환경 변수에 실서비스 상점 키가 노출되지 않도록 토스 테스트용 클라이언트 키와 비밀키를 등록합니다.

  • Frontend (`.env.staging`): 
    VITE_TOSS_CLIENT_KEY=test_ck_... # 토스 상점 테스트 클라이언트 키
  • Backend (Supabase Secrets): 
    TOSS_SECRET_KEY=test_sk_... # 토스 상점 테스트 시크릿 키

2. Toss Payments 웹훅(Webhook) 수신 대상을 Staging으로 연결

카드 등록 완료, 가상 계좌 결제 상태 변화 등을 토스 서버가 우리 백엔드에 즉시 알려주는 웹훅 수신처를 스테이징 API로 정렬합니다.

  • 설정 위치: Toss Payments 개발자 센터 ➔ 테스트(Sandbox) 환경 ➔ 웹훅 설정
  • 웹훅 URL (수신 서버 주소): 
    https://<staging-supabase-project-ref>.supabase.co/functions/v1/payment-webhook
    ※ 실서버(Production) 웹훅과 섞이지 않도록 토스 테스트 상점 계정 관리자에서 수신 URL을 스테이징 엣지 펑션으로 맵핑해주어야 합니다.

3. Grace Period(유예 기간) 및 구독 해지 Edge Function 검증

  • 구독 해지 검사 (`payment-cancel`): 스테이징 Supabase에 배포된 payment-cancel Edge Function이 토스 샌드박스의 빌링키 취소 API를 정상 호출하는지 검사합니다.
  • 유예 기간 검사 시나리오: 테스트를 위해 Staging DB에서 수동으로 구독 레코드의 상태를 변경하여 화면 UI 변화를 검증할 수 있습니다. 
    UPDATE user_subscriptions 
SET status = 'grace_period', grace_period_until = NOW() + INTERVAL '3 days'
WHERE user_id = '<test-user-id>';
주요 세팅 단계

1. Supabase Staging 데이터베이스 초기 배포

로컬에 누적된 마이그레이션 이력을 스테이징 프로젝트로 한 번에 밀어 넣습니다.

# 1. 스테이징 프로젝트 Link
supabase link --project-ref <staging-project-ref-id>

# 2. 로컬 스키마 상태를 스테이징 DB에 배포
supabase db push

# 3. 엣지 함수 배포
supabase functions deploy payment-cancel --project-ref <staging-project-ref-id>
supabase functions deploy payment-webhook --project-ref <staging-project-ref-id>

2. Cloudflare 환경 변수 및 Custom Domain 연동

Cloudflare 대시보드 Pages 프로젝트 설정 ➔ 환경 변수에서 빌드 옵션별로 키를 분기시키고, CNAME에 도메인을 매핑합니다.

  • Staging 환경 변수: barosit-staging의 API URL, Anon Key 및 토스 테스트 클라이언트 키 등록
  • Custom Domain 추가: staging.barosit.com을 등록하여 Preview 배포망에 맵핑

3. QA 러너(runner.mjs) 외부 환경변수 우선 적용 패치

러너가 로컬 Docker(`supabase status`)가 아닌 외부 환경 변수로 접속할 수 있도록 개선합니다.

function getEnv() {
  // 1. 외부 주입 환경 변수가 있으면 즉시 우선 처리
  if (process.env.SUPABASE_URL && process.env.SUPABASE_SERVICE_ROLE_KEY) {
    return {
      url: process.env.SUPABASE_URL,
      anon: process.env.SUPABASE_ANON_KEY,
      svc: process.env.SUPABASE_SERVICE_ROLE_KEY
    };
  }
  
  // 2. 없을 때만 로컬 Docker 커맨드로 자동 파싱 (기존 로직)
  ...
}
