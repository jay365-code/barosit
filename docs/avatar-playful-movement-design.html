1. 배경 & 문제의식

BaroSit의 약점은 재미 요소 부재와 바이럴/신규 유입. 지루한 자세 잔소리 앱이라는 인상을 넘어설 훅이 필요하다.

다행히 BaroSit은 카메라에서 Pose 33점 + 얼굴 메시 468점 + 손 21점 + 인물 세그멘테이션 마스크를 이미 실시간으로 뽑고, 초록 실루엣으로 렌더링까지 한다(SilhouetteOverlay.tsx). 즉 "사람을 스타일화해 그리는 파이프라인"이 이미 존재 — 캐릭터/동물 아바타는 그 자연스러운 확장이다.

제약 상수: 온디바이스 · 카메라 단일 · 추가 모델 도입 최소화(랜드마크 재활용) · 자세 코치 정체성 유지 · 배터리 보수적(opt-in).

2. 핵심 컨셉 — 미션과의 정합

재미 요소가 자세 미션에서 벗어나지 않는다는 것이 이 제안의 핵심. BaroSit의 과학적 근거 자체가 Stuart McGill의 "The best posture is the next posture" — 움직임이 곧 목표다(variabilityTracker.ts). 따라서 "춤/움직임 = 재미"는 미션의 정면 실현이다.

넛지 "일어나 움직이세요" │ ▼ 팬더와 함께 춤/스트레칭 ── 운동 + 재미 │ ▼ 클립 저장 · 공유 ── 바이럴 유입 · 재방문 동기 │ ▼ 더 자주 열고 · 더 많이 움직임 ── 자세·건강 목표 달성 ↑

운동 + 재미 + 공유 + 자세개선이 하나의 선순환으로 도는 구조. 지루한 경쟁 앱이 못 주는 가치.

4. 자세 반응형 마스코트

아바타를 장식이 아니라 "내 자세의 거울"로. 자세 판정 결과를 캐릭터 상태에 매핑한다.

자세/컨디션캐릭터 반응
바르게 앉음활짝 펴짐 · 표정 밝음 · 꽃이 핌
구부정 / 거북목축 처짐 · 시무룩
하품 / 피로(컨디션 레이어)같이 하품 · 눈 처짐
바른자세 스트릭성장 · 의상/액세서리 해금 (보상 넛지 연동)

자세 신호: analyzer.ts / 보상·성장: 기존 넛지 슬라이스(posture-nudge-design.md) 연동

5. 거리 적응형 (앉음 / 섬)

사용자 거리는 기존 신호로 감지 가능(어깨 폭 · noseToShoulderZ · face tz — 캘리브레이션/모니터근접 판정에 이미 사용). 이걸로 아바타 모드를 전환한다.

상황프레임아바타역할
책상 착석(가까움)얼굴·상반신얼굴/상반신 아바타자세 미러링
기립·후퇴(멂)전신 33점전신 캐릭터움직임 미러링·댄스
전신이 잡히는 순간 = 대개 일어나 움직일 때 = BaroSit이 원래 권장하는 행동. "앉으면 자세 거울, 일어서면 움직임 게임"으로 재미와 미션이 맞물린다.

6. 움직임 클립 (팬더 댄스)

아이디어: 재택에서 춤추면 귀여운 팬더가 따라 춤추는 클립을 생성·저장·공유. 모션캡처 → 아바타 → 영상.

Pose 33 (전신) + world landmarks │ [스무딩 · 리타게팅] ▼ 팬더 리그 구동 (canvas 렌더) │ [MediaRecorder — canvas 캡처] ▼ webm/mp4 클립 ── 저장 · SNS 공유
  • 프라이버시 이점: 저장물은 팬더 렌더링뿐, 실제 웹캠 영상 아님 → 안심 공유. 온디바이스 원칙 유지.
  • 마케팅 훅: "내 팬더 댄스" 공유 = 무료 신규 유입 콘텐츠 (SEO/마케팅 방향과 정합).
  • 기술: MediaRecorder로 canvas 스트림 녹화 — 웹 표준, 구현 가볍다.

MVP: "15초 춤춰보세요" → 2D 팬더가 상반신~전신 따라함 → 클립 저장. 정면·잔잔한 동작 기준이면 지금 자산으로 프로토타입 가능.

7. 아바타 꾸미기 · 옷 입히기

아바타 커스터마이징은 재미·보상 루프를 강화한다. 옷 입어보기는 정체성에 맞는 범위까지만 채택.

방식결과온디바이스판정
① 아바타에 코스튬(프리셋/업로드)팬더가 그 옷 입음(스타일화)✅채택 후보
② 실제 몸 2D 오버레이납작한 스티커 느낌✅구매판단엔 조잡
③ 실사 try-on(diffusion)포토리얼❌ 서버 생성AI보류(§11)

결론: 재미 요소로는 ①(아바타 코스튬) 까지가 적정. 정체성·비용상 결이 맞고 공유거리도 된다.

9. 한계 · 리스크

  1. 아트 에셋 = 진짜 비용: 코드보다 캐릭터 디자인·리깅이 큰 작업. 방향(귀여운 동물? 마스코트?)·종수 결정 필요.
  2. 단일 카메라 mocap 한계: 빠른 춤·회전·팔 교차·뒤돌기에서 튀거나 추적 상실. 정면·잔잔한 동작이 최선. 스무딩 필수.
  3. 전신 조건부: 반쯤 앉아 다리 가리면 하체 불안정 → "완전히 물러났을 때만 전신 모드".
  4. 배터리/성능: 부하의 원천은 감지가 아니라 아바타 렌더링. 상세는 아래 성능 프로파일 참고. opt-in · 경량 · 유휴 시 정지 원칙.
  5. 초점 이탈: 정식 기능이 아니라 재미/공유 훅으로. 자세 코치 정체성이 주(主), 아바타는 종(從).

성능 프로파일 — 부하는 감지가 아니라 렌더링

"blendshape(52 표정)를 켜면 메모리/CPU가 확 늘지 않나?" → 아니다. 비용 구조를 분리해 본다.

항목추가 비용이유
얼굴 468점 추론이미 발생 중지금도 pitch/yaw/roll 뽑으려 FaceLandmarker가 매 프레임 추론 — 비싼 부분은 이미 돌고 있음
blendshape 52 출력 ON미미 (~수 %)기존 추론 위에 붙는 작은 출력. init 때 소형 서브모델 1개 + 프레임당 float 52개 → 메모리 무시 수준. 2배로 뛰지 않음
아바타 렌더링진짜 부하리트리버 3D 리그·morph target·canvas 드로우 — 실제 CPU/GPU·배터리를 좌우하는 지점
결론: blendshape는 사실상 공짜에 가깝다. 관리 대상은 렌더링 파이프라인 — opt-in 토글, 경량 렌더(2D 우선), 유휴/일시정지 시 렌더 정지, 프레임레이트 캡으로 배터리 방어. (메모리의 배터리 최적화 방침과 정합)

11. 향후 확장 / 보류 트랙

실사 가상 피팅(virtual try-on): 매력적이나 BaroSit 정체성과 충돌 — 보류 결정.

  • 포토리얼 try-on = diffusion 기반 클라우드 생성AI(IDM-VTON류) → 서버·GPU 비용 + "온디바이스·데이터 안 나감" 약속 파기.
  • 패션 커머스는 별도 산업·시장 — 자세/웰니스와 결이 다르고 초점을 희석.
  • 하려면 온디바이스 원칙·비용 구조를 처음부터 재설계해야 하므로 재미 로드맵과 분리.

12. 열린 질문

  • 마스코트 아트 방향(동물 1종 vs 여러 캐릭터 선택) — 아트 예산과 직결.
  • 아바타를 메인 화면 기본으로 둘지, opt-in 토글로 둘지 (프라이버시 선호 사용자엔 기본 아바타가 이점).
  • 댄스 클립을 무료 vs 유료 차별화 요소로 쓸지.
  • 클립 공유에 워터마크/브랜딩을 넣어 마케팅 훅으로 강화할지.
  • 2D 퍼펫 vs 경량 3D — 초기엔 2D로 검증 후 3D 판단?
