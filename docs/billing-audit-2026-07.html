2026-07-20 · 주식회사 구비드 · 대상: 구독 · 정기결제 · 해지 · 환불 · 웹훅
한 줄 요약 — 정기결제 배치가 등록된 적이 없어 갱신이 한 번도 실행되지 않았고, 결제 원장(billing_history)은 사용자가 직접 수정할 수 있는 상태다. 이 둘이 다른 모든 항목의 전제를 무너뜨린다.
실측프로덕션 DB·스키마에서 직접 확인 코드소스 확인, 런타임 미검증

P0 — 즉시 조치

P0-1실측사용자가 결제 원장을 직접 수정할 수 있음

프로덕션 원격 스키마에서 확인한 실제 정책이다.

CREATE POLICY "Users can update their own billing history"
  ON public.billing_history FOR UPDATE TO authenticated
  USING ((auth.uid() = user_id));          -- WITH CHECK 없음

CREATE POLICY "Users can insert their own billing history"
  ON public.billing_history FOR INSERT TO authenticated
  WITH CHECK ((auth.uid() = user_id));

user_subscriptions 에는 check_subscription_tampering 트리거가 있으나 billing_history 에는 대응물이 없다. 로그인 사용자가 anon 키만으로 PATCH /rest/v1/billing_history 를 호출해 자기 결제 행을 조작할 수 있다.

  • status 를 refunded → completed 로 되돌려 재환불
  • created_at 만 최근으로 바꾸면 7일 청약철회 창이 리셋된다. 미사용 판정도 조작된 시각 이후만 세므로 1년 쓴 연간 구독도 전액 환불
  • amount 를 키워 환불액 팽창

결과적으로 payment-cancel 의 자격 판정이 전부 우회된다.

조치 — INSERT/UPDATE 정책 철회. 원장 쓰기는 Edge Function 의 service_role 로만 이루어지고(클라이언트에는 .select() 경로만 존재) service_role 은 RLS 를 우회하므로 정상 흐름에 영향 없음. SELECT 는 유지.

P0-2실측정기결제가 한 번도 실행된 적 없음

20260521000010_payment_backend.sql:48~ 의 cron.schedule 블록이 전부 주석이다. "service_role key 노출 방지를 위해 마이그레이션에 하드코딩하지 않음 — SQL Editor 에서 1회 실행" 구조인데 그 실행이 누락됐다.

같은 레포의 20260630000001_account_purge_cron.sql:11 은 DO $$ … EXECUTE 로 실제 등록된다. 패턴은 있으나 결제에만 적용되지 않았다.

행동 증거 — 어드민 계정 구독:

current_period_end : 2026-06-28   (21일 경과)
status             : active       ← 강등 안 됨
dunning_attempts   : 0            ← 재시도 흔적 없음
updated_at         : 2026-05-28   ← 결제 이후 아무도 건드리지 않음

대상 선정 쿼리(charge-renewals:30-34)는 정상이다. 호출자가 없을 뿐이다. pg_cron·pg_net 확장은 설치되어 있고, 결제 마이그레이션 13종도 전부 원격 적용 확인.

이번 환불 개편이 만든/활성화한 문제

자체 지적

  1. dryRun 기본값이 파괴적 — req.json().catch(() => ({dryRun:false})) 로 본문 파싱 실패 시 실환불 실행
  2. refunded_amount 덮어쓰기 — payment-cancel 은 대입, admin-refund 는 누적. 어드민 부분환불 후 사용자 셀프 환불 시 누적액 소실
  3. 웹훅이 부분환불 상태를 파괴 — toss-webhook:60 매핑이 DONE→completed, CANCELED→refunded, 그 외 전부 pending. 부분취소 상태가 매핑에 없어 partially_refunded 행을 pending 으로 덮어쓰면 어느 필터에도 안 걸리는 고아 상태가 됨
  4. partially_refunded 를 읽는 코드 0건 — UI(Marketing.tsx:6839)도 completed 만 찾아 부분환불 이력이 사용자에게 안 보임
  5. 부분 환불 후 latest 이동 — completed 만 조회하면 이전 주기 결제가 환불 후보가 됨

payment-cancel v9 는 배포되어 있으나 프로덕션에 연간 결제 건이 0건이라 해당 경로에 아직 도달 불가 — 실질 노출은 없음.
