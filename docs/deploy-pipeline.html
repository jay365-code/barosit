Release & Changelog Pipeline · 협업 관점

BaroSit 배포 파이프라인

실제로 배포는 당신이 저에게 “배포하자” 한 마디로 시작합니다. 그 뒤 제(클로드)가 기계적인 일을 대신 실행하고, 당신은 되돌리기 어려운 두 관문에서만 승인하면, 나머지는 CI가 완전히 자동으로 처리합니다.

당신

결정과 승인. “배포하자” → 문구 승인 → 방출 승인. 되돌리기 어려운 곳만 당신 몫.

클로드 (나)

기계적 실행 대행. 워크플로 실행, CHANGELOG·버전 파일 편집, 커밋, 태그 푸시.

CI (GitHub Actions)

태그 하나에 반응해 빌드·서명·공개·전파까지 사람 손 없이 자동.

🔑
당신만 눌러야 하는 두 관문 — ③ 문구 승인(PR 머지)과 ⑤ 방출 승인(태그). 이 둘은 사용자에게 실제로 나가는 지점이라, 제가 대신 하지 않고 당신 확인을 받습니다.

배포 한 번, 실제 순서

how we do it together
1

배포 결정

당신

저에게 한 마디: “0.9.4 배포하자”. 이게 전체 파이프라인의 시작 신호입니다.

2

릴리스 노트 초안 만들기

클로드

제가 Prepare Release Notes 워크플로를 돌리거나(에이전트 Ethan이 커밋 로그를 사용자 언어로 요약), 필요하면 CHANGELOG를 직접 작성 → 검토용 PR을 준비합니다.

workflow prepare-release.yml secret ANTHROPIC_API_KEY
3

문구 승인

당신

PR의 CHANGELOG 문구를 확인. 고칠 게 있으면 말씀만 하시면 제가 수정, 괜찮으면 머지 = 승인. 사용자에게 보일 문장을 최종 확정하는 곳.

🔒 되돌리기 어려운 관문 ①
4

버전 범프

클로드

제가 앱 매니페스트 4개 파일 버전을 올려 커밋하고 main에 반영합니다. (빠뜨리면 사용자가 업데이트를 못 받으므로 태그 전 필수)

package.json · tauri.conf.json · Cargo.toml · Cargo.lock
5

방출 승인

당신

준비가 끝나면 제가 확인을 요청합니다. “태그 올려” 하시는 순간이 실제 사용자에게 배포가 나가는 지점. 되돌리기 가장 어려운 곳이라 반드시 당신 승인.

🔒 되돌리기 어려운 관문 ②
6

태그 푸시

클로드

승인받으면 제가 태그를 밀어 CI를 깨웁니다. 이 한 줄이 아래 자동 구간의 방아쇠.

$ git tag v0.9.4 && git push origin v0.9.4
7

CI 자동 구간 — 사람 손 없음

CI 자동

태그를 감지한 release.yml이 순서대로 처리합니다:

  • 7a노트 추출 게이트 — CHANGELOG 섹션 추출, 없으면 빌드 전에 실패
  • 7b빌드·서명·공증 — macOS/Windows, 업데이터 서명 + Apple 코드서명·공증
  • 7cRelease 공개 + 자동 업데이트 — Latest로 공개, 기존 사용자에게 배포
  • 7d웹 changelog 게시 — releases 테이블 upsert → #/changelog
8

결과 확인

클로드 + 당신

제가 Actions Summary(✅ 웹 게시)와 barosit.com/#/changelog 반영을 확인해 보고드립니다. 문제 있으면 경고가 뜨니 바로 알 수 있습니다.

안전장치

fail-safe

✓ CHANGELOG 누락 게이트

태그를 밀어도 해당 버전 섹션이 없으면 notes 잡이 빌드 전에 실패 → 빈 릴리스 방지.

✓ 웹 게시 가시화

시크릿 없으면 조용히 넘어가지 않고 경고 + Summary에 “⚠︎ 게시 건너뜀” 표기. 성공 시 기록된 행을 되받아 검증.
