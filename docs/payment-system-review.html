01 한눈에 보는 요약

BaroSit의 결제 시스템은 Toss Payments 빌링(정기결제) 모델을 기반으로, 프론트엔드 SDK → Supabase Edge Functions(서버 신뢰 단일 경로) → Toss API → PostgreSQL 원장으로 이어지는 2-Way Handshake 아키텍처가 비교적 충실하게 구현되어 있습니다. 빌링키 발급·첫 청구·정기청구·더닝(유예)·환불(청약철회)·해지/복구·웹훅 안전망이 모두 코드 레벨에서는 구현 완료 상태이며(docs/launch-readiness-plan.md P0-1 기준), 남은 임계경로는 배포 + Toss 가맹점 심사 + 실결제 E2E QA입니다.

✅ 잘 설계된 점

  • plan/status 변경을 service_role Edge Function으로만 허용하는 DB 트리거 잠금 (위변조 차단)
  • 웹훅 무서명 폴백을 orderId 기반 S2S 교차검증으로 보완
  • order_id 고유 인덱스로 원장 멱등성 확보
  • 결제 실패 시 즉시 박탈 대신 7일 유예(grace_period) 더닝
  • raw 카드 미수집 — Toss 호스티드 결제창 위임으로 PCI 범위 최소화

⚠️ 핵심 보완 필요

  • 치명 Free 계정이 Pro 혜택 사용 — 계정전환 시 전역 플랜 캐시 잔존 (§7 E1·E2)
  • 치명 첫 결제 성공 후 DB 저장 실패 시 보상 트랜잭션 부재 (돈만 빠짐)
  • 치명 웹훅 서명·호출자 검증 없음 (재전송 무한루프·DoS)
  • 높음 canceled 만료 후 plan_id='pro' 잔존 정리 로직 없음
  • 높음 사용자 결제실패/유예 알림 메일·푸시 미발송
  • 중간 카드정보 표시 필드명 버그(brand↔company)
Stripe 관점 현재 코드는 _shared/toss.ts에 Toss API가 직접 강결합되어 있고, billingVerify.ts에 Stripe 서명검증 함수가 작성돼 있으나 어떤 함수도 사용하지 않는 dead code입니다. Stripe 추가 전에 PaymentProvider 추상화 레이어와 DB provider 컬럼 도입이 선행되어야 합니다(§8).

03 결제 라이프사이클 흐름

① 신규 가입 결제 (빌링키 발급 + 첫 청구)

06 점검 결과 — 보완 사항

심각도: 치명 금전·보안 직접 영향 · 높음 운영 신뢰성 · 중간 정합성·UX · 낮음 정리·유지보수

치명 C1. 첫 결제 성공 후 DB 저장 실패 시 보상 트랜잭션 부재

billing-issue/index.ts:64-91 — chargeBilling 성공 후 user_subscriptions.upsert 실패 시

chargeBilling()으로 실제 카드 청구는 완료됐는데 이후 user_subscriptions.upsert 또는 billing_history.insert가 실패하면, 사용자는 돈은 빠져나갔는데 PRO 활성화가 안 되는 상태가 됩니다. 현재 catch는 단순히 400 에러만 반환하고 청구를 되돌리지 않습니다. 프론트는 1200ms 후 1회만 호출하므로 재시도도 없습니다.

보완: 첫 청구 성공 후 DB 쓰기 실패 시 cancelPayment(paymentKey)로 즉시 자동 환불(보상)하거나, "결제는 됐으나 활성화 지연" 상태로 원장에 pending 적재 후 별도 reconcile 잡으로 복구. 최소한 paymentKey를 로그/알림에 남겨 수동 환불 가능하게 할 것.

치명 C2. 웹훅 서명·호출자 검증 부재 + 재전송 무한루프

toss-webhook/index.ts (verify_jwt=false) · config.toml:32

웹훅은 인증이 없어 누구나 호출 가능합니다. orderId만 넣으면 매 요청이 getPaymentByOrderId(Toss API 호출)을 트리거하므로 외부 발 Toss API 호출 폭주(DoS·요율한도 소진)가 가능합니다. 또한 ledger 조회가 영구적으로 실패하면 항상 500을 반환해 Toss 재전송이 무한 반복됩니다. billingVerify.ts에 Toss 서명검증이 아닌 S2S 함수만 있고, 실제 Toss-Signature 헤더 HMAC 검증은 없습니다.

보완: ① Toss 웹훅 서명(Toss-Signature/IP 대역) 검증 추가 ② ledger 조회 N회 실패 시 200 + 알림으로 전환해 무한 재전송 차단 ③ 알 수 없는 orderId는 조기 200 반환해 API 호출 자체를 회피.

07 권한(Entitlement) 적용 점검 — Free↔Pro 누수

"결제하면 Pro 가입자가 되고 Pro 혜택을 받는다"는 명제의 역방향 — 즉 결제하지 않은 Free 사용자가 Pro 혜택을 받게 되는 경로를 별도 점검했습니다. 결론부터: 실제로 새는 경로가 여럿 존재하며, 사용자가 관찰한 "Pro 계정으로 로그인했다가 Free 계정으로 로그인해도 앱(모니터링)이 계속 동작" 현상은 코드상 재현 가능한 실제 버그입니다.

권한 판정 데이터 흐름

[진실원] DB user_subscriptions (plan_id, status, current_period_end) │ resolveEffectivePlan() ← isBetaFree()면 row 무관 'pro' 반환 ▼ [캐시] localStorage['barosit:subscription_plan'] ◀── ⚠️ 전역 키(계정 비귀속, 사용자 편집 가능) │ (동기 read — 부팅 즉시) ├──▶ useMonitoringEngine.subPlan │ enabled = ... && (subPlan === 'pro' || opts.visible) ← 백그라운드 모니터링 게이트 │ └──▶ syncService.userPlan if (userPlan !== 'pro') return ← 서버 동기화 게이트 [교정] PricingView / Marketing 진입 시에만 DB 대조 → 위변조 시 'free' 롤백 + critical 알림 ✗ App.tsx(데스크톱 메인) · useMonitoringEngine 은 교정 안 함
핵심 결함 엔트리먼트의 진실원은 DB인데, 실제 기능 게이트(모니터링·동기화)는 계정에 귀속되지 않은 전역 localStorage 캐시를 직접 읽습니다. 이 캐시는 ① 계정 전환 시 갱신되지 않고 ② 다운그레이드가 반영되지 않으며 ③ 사용자가 직접 편집 가능합니다.

치명 E1. 계정 전환 시 이전 Pro 캐시 잔존 → Free 계정이 Pro 사용 (사용자 보고 현상)

useAuth.ts:431-440 (signOut) · Marketing.tsx:5297 (handleSignOut) · ProfileView.tsx:977

barosit:subscription_plan은 user_id로 네임스페이스되지 않은 단일 전역 키입니다. useAuth.signOut()은 dispatchAuthSync(null) + supabase.auth.signOut()만 호출하고 이 캐시를 비우지 않습니다. 웹 handleSignOut도 동일합니다. 오직 ProfileView의 로그아웃 버튼만 명시적으로 'free'로 세팅합니다.

따라서 재현 경로:

1. Pro 계정 로그인 → DB 조회 → localStorage['..._plan'] = 'pro'
2. 로그아웃 (ProfileView 버튼 외 경로) → 캐시 'pro' 그대로 잔존
3. Free 계정 로그인 → 캐시는 여전히 'pro'
4. useMonitoringEngine 이 캐시 'pro' 를 읽음 → Free 계정에 Pro 백그라운드 모니터링 부여
보완: ① 로그인 시점에 캐시를 무조건 제거하고 DB 재조회로만 재설정 ② signOut을 공통화해 항상 캐시 제거 ③ 또는 키를 barosit:subscription_plan:{userId}로 네임스페이스(공용 PC 대비).

치명 E2. 다운그레이드가 캐시에 write-back 되지 않음 (데스크톱 메인 경로)

App.tsx:385-420 (fetchSub) — setSubPlan만, localStorage.setItem 없음

데스크톱 앱의 메인 구독 해석기인 App.tsx의 fetchSub()는 DB에서 resolveEffectivePlan(data)로 plan을 구해 React state(setSubPlan)만 갱신하고 localStorage에는 절대 쓰지 않습니다(읽기만 4곳). 반면 실제 게이트인 useMonitoringEngine·syncService는 localStorage를 직접 읽습니다.

결과: DB상 free로 강등(해지 만료·환불·더닝 강등)되어도 localStorage의 옛 'pro'가 남아 모니터링 엔진은 계속 Pro로 동작합니다. barosit:subscription-changed 이벤트를 받아 재read해도 캐시값 자체가 안 바뀌었으므로 무의미합니다. PricingView/Marketing만 캐시를 교정하므로, 해당 화면을 거치지 않는 한 영구히 어긋납니다.

보완: fetchSub가 해석된 plan을 localStorage.setItem으로 동기화하거나(단일 진실원), 모든 게이트가 localStorage 대신 중앙 구독 상태(혹은 부팅 시 DB 재확인)를 읽도록 통일.

치명 E3. 클라이언트 신뢰 게이팅 — 변조로 Pro 획득 가능 (구조적)

useMonitoringEngine.ts:116-126, 340 · syncService.ts:35

Pro의 핵심 가치인 로컬 자세 모니터링은 전적으로 클라이언트에서 동작하며 서버 검증이 없습니다. 게이트가 읽는 localStorage는 사용자가 devtools에서 즉시 바꿀 수 있습니다: 

localStorage.setItem('barosit:subscription_plan','pro');
window.dispatchEvent(new Event('barosit:subscription-changed'));
// → 결제 없이 Pro 백그라운드 모니터링 활성화

위변조 감지(tampering detection)는 PricingView/Marketing 진입 시 DB 대조 시점에만 작동하고, 모니터링 엔진 자체는 무검증입니다. 데스크톱 앱은 PricingView를 안 거치고도 모니터링을 쓸 수 있습니다.

✅ 구현됨: 서버 재검증 훅 useEntitlement 신규 — 모니터링/위젯 게이트가 localStorage 가 아닌 서버 user_subscriptions 재조회 결과(in-memory)를 신뢰. 마운트·포커스·10분 주기·구독변경 시 재검증, 캐시=PRO·서버=FREE 면 자동 강등 + tampering_detected 경보, 오프라인 14일 초과 미검증 시 보수적 FREE. (캐시는 오프라인 시드로만 사용 → devtools 조작은 다음 검증에서 덮어써짐) · Tauri 런타임 QA 필요.
추가 발견 (데스크톱) 추적 결과 백그라운드/위젯 모니터링은 plan으로 전혀 강제되지 않는 상태임이 확인됨: ① App.tsx:277의 구독 상태는 _subPlan(미사용)이고 ② 메인 최소화 시 위젯 전환(App.tsx:513 switchToWidgetMode)은 plan 미확인이며 ③ Widget이 엔진에 visible:true(Widget.tsx:237)를 하드코딩해 유일한 plan 게이트(useMonitoringEngine.ts:340 subPlan==='pro' || visible)를 상시 우회. 결과적으로 Free/게스트도 최소화하면 백그라운드 관제를 받는 누수가 있었다.
→ 해결됨: 제품 결정 해석 B(앱=로그인+Pro 전용)에 따라 포그라운드(MonitorView.tsx:1079)와 백그라운드/위젯(useMonitoringEngine.ts:340, opts.visible 바이패스 제거) 모두 !!user && subPlan==='pro'로 게이팅. (위 "로그아웃 권한 회수 정책" 표 참조 / Tauri 런타임 QA 필요)

08 Stripe 추가 설계 고려사항

상태: 보류 Stripe는 아직 가맹점 신청 전이라 본 점검 보완 범위에서 제외했다. 아래는 신청 후 착수할 설계 참고용이며, 현재 코드에는 미반영이다.

Stripe를 추가하면 두 PG가 근본적으로 다른 정기결제 모델을 갖는다는 점이 핵심 설계 난점입니다. Toss는 우리가 빌링키를 들고 cron으로 직접 청구하는 반면, Stripe는 Stripe Subscription 객체가 청구를 주관하고 우리는 invoice 웹훅을 수신합니다.

관심사현행 TossStripe 추가 시
정기청구 주체우리 cron(charge-renewals)이 능동 청구Stripe Subscription이 자동 청구 → cron 대상에서 제외 필요
결제수단 등록requestBillingAuth → authKey → 빌링키SetupIntent / PaymentMethod / Checkout Session
더닝직접 구현(grace 7일)Stripe Smart Retries + dunning 내장 → 중복 구현 주의
환불cancelPayment(paymentKey)Refund API (paymentIntent)
웹훅 검증S2S 재조회(서명 없음)Stripe-Signature HMAC (billingVerify에 작성됨, 미연결)
통화KRW 정수 고정다통화(USD/JPY) — i18n ko/en/ja와 연동, 금액 모델 재설계
고객/카드 관리 UI자체 ProfileViewStripe Customer Portal 위임 가능(자체 UI 부담↓)
멱등성orderId 고유 인덱스Idempotency-Key 헤더 표준

권장 추상화 방향

// _shared/payment/provider.ts (신규)
interface PaymentProvider {
  issueRecurringMethod(...): Promise<{ token; card }>   // Toss billingKey | Stripe paymentMethod
  charge(...): Promise<ChargeResult>                     // Toss /billing | Stripe PaymentIntent
  refund(paymentRef, amount?): Promise<RefundResult>
  verifyWebhook(req): Promise<{ valid; event }>
}
// toss.ts → class TossProvider implements PaymentProvider
// stripe.ts(신규) → class StripeProvider implements PaymentProvider
선행 필수 ① DB에 user_subscriptions.provider · billing_history.provider 컬럼 추가(없으면 두 PG 원장이 섞임) ② billing-issue/charge-renewals/payment-cancel을 provider 분기로 리팩터 ③ charge-renewals가 Stripe 구독은 청구하지 않도록 provider 필터 ④ stripe-webhook Edge Function 신규(invoice.paid / payment_failed / subscription.deleted 처리) ⑤ resolveEffectivePlan은 provider 무관하게 동작하므로 유지 가능.
