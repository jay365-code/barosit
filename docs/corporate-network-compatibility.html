01 한눈에 보는 요약

BaroSit의 핵심 가치인 "자세 감지·코칭"은 온디바이스(로컬) 추론이라 사내망에서도 동작합니다. 단, 자세 감지 AI 모델(MediaPipe)을 최초 1회 CDN에서 내려받아야 하고, 로그인·클라우드 동기화·구독 결제·자동 업데이트는 모두 외부 SaaS 도메인에 대한 HTTPS/WebSocket 연결이 필수입니다. 보안이 철저한 사내망(아웃바운드 화이트리스트 정책, SSL 인스펙션, WebSocket 차단)에서는 아래 기능들이 깨질 수 있습니다.

✅ 사내망에서도 동작 (로컬)

  • 자세·목/어깨/손 감지 추론 (온디바이스, 카메라 영상은 외부 전송 없음)
  • 실시간 자세 경고·넛지·집중 모드 (로컬 로직)
  • 로컬 통계/설정 (로그인 없이 사용 가능한 범위)
  • 단, AI 모델 파일은 최초 1회 다운로드 필요(이후 캐시)

⚠️ 사내망 차단 시 실패

  • 치명 소셜 로그인 (Google/Kakao/Apple/Naver/Line OAuth)
  • 치명 클라우드 동기화·Realtime (Supabase, WebSocket)
  • 치명 구독 결제 (Toss Payments)
  • 높음 자세 AI 모델 최초 다운로드 (Google Storage/jsDelivr)
  • 중간 자동 업데이트 (GitHub Releases)
  • 낮음 LLM 코칭 문구·문의 메일·웹폰트 (폴백 있음)
전제 — 망 정책은 바뀌지 않는다 현실적으로 고객사가 BaroSit 하나 때문에 방화벽·망 정책을 변경하지 않습니다. 따라서 "IT에 화이트리스트(§7)를 요청하라"는 것을 주(主)전략으로 삼아선 안 됩니다. §7은 IT가 협조 가능한 드문 경우의 참고 자료일 뿐이고, 우리의 실제 대응은 아래 두 가지입니다.
핵심 전략 ① 원인 투명 안내 — 막히면 "왜 안 되는지"를 사용자에게 명확히 설명해 막연한 이탈을 막는다(§8 R2). ② 제한적 서비스 연속성 — 이미 이용 권한이 있는 사용자(Pro·무료체험)는 네트워크가 막혀도 자세 감지·경고가 끊기지 않도록 오프라인 회복력을 보장한다(§8 R1·R3). 동기화·신규 결제는 잠시 포기하더라도 핵심 효용은 유지한다. (자세 감지는 Pro 전용 정책 유지 — 비로그인/Free 에게 무료 개방하는 것이 아니라, 정당한 Pro 가 차단망에서 강등되지 않게 하는 것)

적용 범위 — 환경별 3단계 대응

"망 변경 없이도 무엇을 제공할 수 있는가"를 기준으로, 고객 환경을 3단계로 나눠 대응합니다.

환경대응제공 범위
① 일반(개인·재택·일반 사무망)
인터넷 가능, 카메라 허용		 전체 제공 로그인·동기화·구독 포함 모든 기능 정상
② 보안 이슈 사업장
외부망 일부/전면 차단, 단 카메라는 사용 가능		 제한적 연속성 이미 Pro·무료체험인 사용자는 오프라인/차단 환경에서도 자세 감지·경고가 끊기지 않음(강등 안 함). 신규 결제·클라우드 동기화·자동 업데이트만 보류되고, 원인은 사용자에게 안내. (자세 감지는 Pro 전용 — 비로그인/Free 게스트에게 열리는 것은 아님)
③ 고보안 / 망분리

설계 원칙 망 변경을 전제하지 않는다. ②번 "보안 이슈 사업장"이 핵심 공략 지점 — 망을 못 바꿔도 카메라만 되면 로컬 핵심 기능이 끊기지 않게 만들어, 막힌 환경에서도 제한적 서비스 연속성을 제공합니다. ③번(망분리·카메라 금지)만 현 단계 타겟에서 제외하며, 별도 제품 라인(오프라인 빌드·온프레미스 라이선스·보안성 심의)이 필요한 시장으로 남겨둡니다.

05 차단 시나리오별 동작

시나리오 A — 전면 화이트리스트(허용 목록 외 전면 차단), BaroSit 미등록

[앱 실행] ├─ 모델 다운로드 시도 → storage.googleapis.com / jsdelivr ❌ 차단 │ → 캐시 있음: 자세 감지 정상 / 캐시 없음: 감지 시작 실패 ├─ 로그인 시도 → Supabase + OAuth provider ❌ 차단 → 로그인 불가 ├─ 동기화/Realtime → *.supabase.co (WSS) ❌ 차단 → 동기화 멈춤 └─ 결제 → tosspayments.com ❌ 차단 → 결제 불가 [결과] 최초 실행이면 사실상 사용 불가 / 이미 모델 캐시·로그인된 상태면 "로컬 자세 감지"만 동작

시나리오 B — 주요 SaaS는 허용, 소셜·결제 카테고리만 차단

회사가 클라우드 스토리지·API는 허용하되 "소셜미디어"/"결제" 카테고리를 DNS로 차단하는 경우. 로그인이 막히면 BaroSit은 비로그인 로컬 모드로만 쓸 수 있습니다.

시나리오 C — 인증 프록시 + SSL 인스펙션

가장 까다로운 환경. 도메인은 허용돼도 (1) 프록시 인증을 통과하지 못하거나, (2) 사내 루트 CA를 클라이언트가 신뢰하지 않으면 TLS 핸드셰이크가 깨집니다. 특히 WebSocket(Realtime)과 Tauri Rust(reqwest) 호출(LLM·업데이터)은 OS 신뢰스토어/프록시 설정을 따라가지 못할 위험이 큽니다(§6).

시나리오 D — 권장 화이트리스트 적용(§7)

§7 목록을 443/WSS로 허용하면 모든 기능 정상 동작. 가장 깔끔한 해법이며 IT 부서에 §7 표를 그대로 전달하면 됩니다.

06 특수 통제별 위험 (SSL 인스펙션·WebSocket·딥링크)

① WebSocket (Supabase Realtime)

동기화·실시간 알림은 wss://*.supabase.co로 연결합니다. 방화벽이 HTTP는 통과시켜도 WebSocket Upgrade를 별도로 차단하거나 SSL 인스펙션이 WS 프레임을 끊는 사례가 흔합니다. 이 경우 로그인·1회성 조회는 되는데 실시간 갱신만 안 되는 부분 장애로 나타납니다.

② 커스텀 URI 스킴 딥링크 (barosit://)

데스크톱 OAuth는 외부 브라우저에서 인증 후 barosit.com/desktop-auth-redirect.html → barosit://auth-callback로 앱에 복귀합니다. 브라우저 정책이 커스텀 스킴 실행을 차단하면 로그인이 브라우저에서 멈춥니다. Windows는 루프백(127.0.0.1 고포트) 폴백이 있으나, 이 역시 로컬 포트 정책에 막힐 수 있습니다.

③ SSL 인스펙션 & 사내 루트 CA

BaroSit은 인증서 핀닝을 쓰지 않으므로, OS 신뢰스토어에 사내 루트 CA가 설치돼 있으면 대부분의 HTTPS는 통과합니다. 다만 Tauri의 Rust 측 HTTP 클라이언트(업데이터·LLM)가 OS 신뢰스토어/프록시를 자동 반영하는지는 환경별 검증이 필요합니다(§8 R4).

④ 카메라 접근 정책 (EDR/DLP)

사내 보안 솔루션이 카메라 접근을 차단/감사하면 자세 감지 자체가 동작하지 않습니다. BaroSit은 카메라 영상을 외부로 전송하지 않고 로컬에서만 추론하므로 이 점을 IT 부서에 명확히 안내하면 승인에 유리합니다(§9).

07 IT 관리자용 허용(Allowlist) 목록

참고용 자료입니다 앞서 밝혔듯 대부분의 고객사는 BaroSit 때문에 망 정책을 바꾸지 않습니다. 이 목록은 IT가 협조 가능한 드문 경우(소규모·유연한 사내망)에 한해 그대로 전달할 수 있는 참고 자료이며, 주 전략은 §8 R1·R2·R3의 제한적 서비스 연속성입니다.

08 제품 개선 권고

화이트리스트 협조가 어려운 기업 고객까지 확보하려면, 사내망 친화성을 높이는 다음 작업을 권고합니다.

✅ 완료 R1. 자세 AI 모델 앱 번들링(오프라인 우선)

구현 완료(커밋 d97037f, 2026-06-26). MediaPipe WASM·모델 4종을 CDN(jsdelivr·storage.googleapis.com) 런타임 다운로드에서 앱 동봉 정적 자산(/mediapipe/...)으로 전환. CDN 차단·오프라인 환경에서도 자세 감지 코어가 동작하고, 자리비움 복귀 시 재다운로드도 사라짐 — 사내망 도입의 가장 큰 진입장벽 제거.

  • scripts/vendor-mediapipe.mjs: wasm은 node_modules에서 복사, 모델은 1회 다운로드 → public/mediapipe/ (멱등). predev/prebuild 훅 자동 실행.
  • src/pose/detector.ts: 로컬 경로 로딩, src 내 CDN 참조 0건. 번들 자산 약 54MB(빌드 시 dist 동봉, 레포는 .gitignore 제외).
  • 잔여: 재빌드 후 실런타임(카메라+자리비움 복귀) 동작 확인.

✅ 일부 완료 R2. 차단 원인 자동 진단·안내 (사용자 자력 대응)

✅ 1차 구현: 동기화 상태가 offline/error일 때 설정의 클라우드 동기화 섹션에 원인 안내 문구를 노출 — "사내 방화벽·보안 정책이나 네트워크 문제로 동기화 서버에 연결하지 못할 수 있으며, 자세 감지·경고 등 핵심 기능은 계속 사용 가능하고 복구 시 자동 동기화"(ko/en/ja i18n). src/views/SettingsDrawer.tsx, src/i18n/locales/*/settings.json (sync.networkNote). 잔여: 도메인별 능동 진단(probe)·로그인 실패 시 안내·게이트별 세분 메시지.

보안 때문에 기능이 막힌 사용자가 "그냥 안 돼요"로 이탈하지 않게, 막힌 원인을 솔직히 설명하고 "그래도 쓸 수 있는 것"으로 안내하는 것이 핵심입니다. 망 변경을 요구하는 게 아니라, 차단을 전제로 한 안내입니다.

동작 방식

  • 로그인·동기화·모델 다운로드 실패를 감지하면, 핵심 도메인 도달 여부를 점검(Supabase / OAuth / Toss / 모델 CDN / WebSocket).
  • 막연한 에러 대신 원인별 메시지 + 무엇이 영향받는지 안내: 예) "회사 보안 정책이 실시간 동기화·로그인 연결을 차단하고 있어요. 클라우드 동기화와 구독은 사용할 수 없지만, 자세 감지·자세 경고는 그대로 사용할 수 있습니다."
  • WebSocket만 끊긴 경우("로그인은 됐는데 실시간 갱신 안 됨") 같은 부분 장애도 구분해 안내.
  • 차단 시 1순위 행동은 "로그인 없이 자세 감지 계속하기"(게스트/오프라인 모드, R3) — 즉시 핵심 효용으로 연결.
  • (보조) IT 협조가 가능한 사용자를 위해 §7 허용 목록을 복사해 전달할 수 있는 옵션만 제공 — 강요하지 않음.

효과: 망이 막혀도 사용자는 "왜 안 되는지 + 무엇은 되는지"를 알고 계속 사용 → 보안 이슈 사업장에서 이탈 방지, 지원 문의 감소.

✅ 일부 완료 R3. 오프라인 Pro 회복력 = 제한적 서비스 연속성의 본체

"보안 이슈 사업장"(환경 ②) 공략의 핵심. 정당한 Pro·무료체험 사용자가 차단망/오프라인에서 권한 확인이 안 돼도 자세 감지가 끊기지 않게 하는 것. 자세 감지는 Pro 전용 정책을 유지하되(비로그인/Free 게스트 개방 아님), 서버 재검증 실패를 강등 사유로 쓰지 않는다. R1(모델 번들)과 합쳐야 완성 — 모델까지 로컬에 있어야 진짜 오프라인이 되므로.

권한(Entitlement) 처리 — 결정 사항 (오프라인 회복력 = A안 범위)

  • ✅ 구현: 오프라인/조회 실패 시 과거 서버 검증 이력이 있는 Pro 는 강등하지 않음(기존 14일 캡 제거). 로그인 자체가 온라인을 요구하므로 정당한 Pro 는 검증 이력 보장 → 차단망에서 무기한 유지. src/auth/useEntitlement.ts
  • ✅ 구현: 무료체험(beta_free) 모드는 오프라인에서도 Pro 와 동일 처리.
  • ✅ 유지: 검증 이력이 전혀 없으면(캐시 신뢰 불가) Free — 변조 방어. 온라인 복구 시 서버값으로 자동 정정(해지/환불 강등 정상 동작).
  • 네트워크 필수인 것(신규 결제·클라우드 동기화·자동 업데이트)만 보류하고 그 이유를 안내(R2).
  • ※ 당초 광범위 C안("코어를 전원 무료")은 자세 감지가 곧 Pro 유료상품이라 수익모델과 충돌 → 오프라인 회복력(A안 범위)으로 좁혀 채택.

P2 R4. 프록시 / 사내 루트 CA 대응

GitHub Releases 차단 기업을 위해 MSI/사일런트 배포 패키지와 사내 업데이트 서버(또는 자동 업데이트 비활성 옵션) 제공. MS Store 채널도 대안.

09 사용자·영업용 안내 체크리스트

사내망에서 "로그인/동기화가 안 돼요" 1차 점검

  1. 같은 PC에서 개인 핫스팟/모바일 테더링으로 바꿔 로그인 시도 → 되면 사내망 차단이 원인.
  2. 브라우저에서 https://kllcnllkcewnutxodwhx.supabase.co 접속 확인 → 막히면 Supabase 차단.
  3. 로그인 시 브라우저가 열리고 멈추면 → 커스텀 스킴(barosit://) 차단 가능성.
  4. 로그인은 되는데 동기화만 멈추면 → WebSocket 차단 가능성.
  5. 차단이 확인되면 망 변경을 기다리지 말고 → "로그인 없이 자세 감지 계속하기"(오프라인 모드)로 사용. 동기화·구독만 보류될 뿐 핵심 기능은 유지됩니다.
  6. (선택) IT 협조가 가능한 환경이면 §7 표를 전달해 허용 요청 — 필수 아님.

IT 부서 승인 요청용 핵심 메시지

  • 카메라 영상은 외부로 전송되지 않습니다. 자세 분석은 100% 사용자 PC 내 온디바이스 추론입니다.
  • 서버로 가는 데이터는 계정/설정/통계 동기화 등 비영상 메타데이터뿐입니다.
  • 필요한 허용 목록은 §7이며, 모두 표준 HTTPS(443)/WSS입니다. 비표준 포트·프로토콜은 사용하지 않습니다.
  • 완전 오프라인 사내 배포가 필요하면 모델 번들 빌드(R1) 제공이 가능합니다(로드맵).
