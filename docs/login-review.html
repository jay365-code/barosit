핵심 결론

macOS 앱과 Windows 앱의 소셜 로그인은 현재 구조상 절대 성공할 수 없습니다. Tauri 코드 경로가 OAuth 팝업을 https://barosit.com(웹 origin)으로 보내고, 메인 앱은 tauri://localhost(다른 origin) 의 자기 localStorage만 polling 합니다. 두 origin 의 localStorage 는 브라우저 표준상 격리되어 있어, 팝업에 세션이 만들어져도 메인은 영원히 찾지 못합니다. 폴링은 10분 후 조용히 종료되고 사용자는 무한 로딩만 봅니다. 웹은 PKCE 표준 경로라 동작합니다.

현재 UI 에 노출된 provider 는 Google + Kakao 두 개뿐입니다. signInWithApple, signInWithNaver, signInWithLine 은 useAuth.ts 에서 export 만 되어 있고 어떤 화면에서도 호출되지 않는 dead export 입니다. Naver/Line 은 그 자체로 Supabase 가 지원하지 않는 provider 라 as any cast 로 타입 검사를 우회한 상태입니다.

3. 아키텍처 개요 — 두 갈래 flow

웹 은 표준 PKCE — signInWithOAuth 가 같은 탭을 provider 페이지로 full-page redirect, provider 가 https://barosit.com/#/auth/callback 으로 돌려보냄. detectSessionInUrl: true 가 코드를 exchange 하고 AuthCallback 이 보조로 한 번 더 시도. (src/auth/supabase.ts:20-29, src/web/Marketing.tsx:779-832) — 표준대로면 문제없음.

Tauri (Mac/Win) 은 implicit flow 로 시도 — signInWithOAuth({ skipBrowserRedirect: true }) 로 authorize URL 만 받아서 새 WebviewWindow("oauth-login") 에 띄움. 메인 앱은 setInterval 로 매 1초 supabase.auth.getSession() 폴링. 세션 잡히면 팝업 닫고 메인 window.location.reload(). (src/auth/useAuth.ts:54-132)

┌─────────────┐    spawn popup     ┌──────────────────────┐
│  main app   │ ─────────────────► │  oauth-login popup   │
│ tauri://    │                    │  https://accounts... │
│ localhost   │                    └──────────┬───────────┘
│             │                               │ OAuth 완료
│ getSession  │                               ▼
│  polling    │                    ┌──────────────────────┐
│  every 1s   │ ◄─ ❌ origin 격리   │  https://barosit.com │
│             │     localStorage    │  /#/auth/callback     │
└─────────────┘     찾을 수 없음    │  session 저장 (여기) │
                                   └──────────────────────┘

이 다이어그램이 CRIT-1 의 본질입니다. session 은 팝업의 https://barosit.com origin localStorage 에 들어가고, 메인 앱의 supabase client 는 tauri://localhost origin localStorage 만 읽음. 둘은 영구히 만나지 않습니다.

4. 치명적 결함 (Critical)

CRITICALCRIT-1

Tauri OAuth 가 메인 앱에 세션을 전달할 수 없음 (Mac + Windows 양쪽 모두 깨짐)

파일 · src/auth/useAuth.ts:54-132

원인: 팝업이 https://barosit.com/#/auth/callback 에 착륙 → 세션이 barosit.com origin 에 저장됨. 메인 앱은 tauri://localhost (또는 http://tauri.localhost) origin 의 supabase client 가 자기 origin 의 localStorage 만 읽음. 브라우저 표준상 origin-partitioned 이라 메인은 영원히 세션을 볼 수 없음.

증상: 사용자가 로그인 버튼을 누르면 팝업이 뜨고 OAuth provider 페이지에서 로그인까지는 성공함. 팝업이 barosit.com 으로 돌아간 뒤 멈춤 (팝업의 AuthCallback 은 window.location.replace("#/app") 로 자기 origin 의 /app 페이지로 이동만 함). 메인 앱은 calling loop 만 돌리다가 10분 timeout 으로 조용히 종료.

증거:

// src/auth/useAuth.ts:65-74
const { data, error } = await supabase.auth.signInWithOAuth({
  provider,
  options: {
    redirectTo: "https://barosit.com/#/auth/callback",  // ← 웹 origin
    skipBrowserRedirect: true,
    ...
  },
});

// src/auth/useAuth.ts:99-110
const checkInterval = setInterval(async () => {
  const { data: { session } } = await supabase.auth.getSession();  // ← tauri://localhost 의 storage
  if (session?.user) { ... }
}, 1000);

올바른 해법 (택일):

  1. tauri-plugin-deep-link 도입 + 커스텀 스킴 (예: barosit://auth-callback) 등록. provider 에서 barosit:// 로 직접 돌려보내고, Rust 측 deep-link handler 가 토큰을 받아 main webview 의 supabase client 에 setSession() 으로 주입. 업계 표준 — 가장 권장.
  2. 로컬 HTTP loopback 서버 를 Rust 측에서 띄우고, redirect URL 을 http://127.0.0.1:/callback 으로 설정. callback 받으면 IPC 로 main 에 전달. (Google CLI tools 가 쓰는 방식.)
  3. 팝업 webview 와 main webview 의 origin 통일 — 팝업도 tauri://localhost/#/auth/callback 으로 redirect. 그러나 Supabase/Google/Kakao 가 tauri:// redirect 를 허용하지 않으므로 현실적으로 불가.
영향: macOS 앱, Windows 앱 / 현재 사용자 영향: 데스크탑 앱에서 로그인 자체가 불가
CRITICALCRIT-2

Naver / Line provider 가 Supabase 미지원인데 as any 로 우회

파일 · src/auth/useAuth.ts:157-181

Supabase JS SDK v2 의 Provider union 에 naver, line 은 없음. 코드는 provider: "naver" as any 식으로 타입 검사를 우회. 호출되면 Supabase 서버가 "Unsupported provider" 로 4xx 반환.

현재 다행: UI 에서 호출하는 곳이 없음 (dead export). 단 export 되어 있어 외부에서 사용 가능한 상태.

해법: Naver/Line 을 정말 지원할 거면 Supabase Custom OIDC Provider 로 등록 + provider 측 client 설정 필요. 아니라면 dead code 제거.

영향: 전 클라이언트 / 현재 사용자 영향: 0 (UI 미노출). 다만 향후 잘못 노출되면 즉시 사용자 에러.
CRITICALCRIT-3

Tauri Apple OAuth 는 일반 OAuth 와 다른 요건을 충족 못 함

파일 · src/auth/useAuth.ts:54 (provider union 에 "apple" 포함)

Apple Sign In 은:

  • Services ID 와 native App ID 가 다름. 데스크탑에서 OAuth 흐름으로 가려면 Services ID + HTTPS redirect 필요 (Apple 정책).
  • prompt=select_account 가 무의미하고 query param 처리도 다름 (코드는 google/kakao 만 분기 — 이건 OK).
  • CRIT-1 의 origin 격리 문제는 동일.

현재 다행: UI 에 Apple 버튼이 없어 호출 불가.

영향: Tauri 양쪽 / 현재 사용자 영향: 0 (UI 미노출).

5. 중대 결함 (High)

HIGHHIGH-1

Tauri 팝업이 자동으로 닫히지 않음

파일 · src/auth/useAuth.ts:99-110

팝업 닫힘 조건이 메인의 polling 이 세션을 잡은 다음. CRIT-1 때문에 영영 안 잡힘 → 팝업은 사용자가 수동으로 닫을 때까지 표시. alwaysOnTop: true 라 다른 작업을 가림.

영향: Mac / Windows
HIGHHIGH-2

ProfileView 의 "popup sandbox" auto-trigger 는 unreachable + 위험

파일 · src/views/ProfileView.tsx:175-196

URL 에 ?is_popup=true&provider=google 가 있으면 자동으로 signInWithGoogle() 호출. 그런데 useAuth.ts 가 팝업을 띄울 때 그 파라미터를 절대 붙이지 않음. 따라서 정상 흐름에서 도달 불가. 단 누군가 URL 을 조작해서 들어오면 Tauri 팝업 안에서 또 팝업을 spawn 하는 무한 재귀 위험. 죽은 코드 + 잠재적 푸트건.

영향: Mac / Windows / 현재 사용자 영향: 0 (도달 경로 없음). 코드 단순화 차원에서 제거 권장.
HIGHHIGH-3

Tauri redirectTo 가 barosit.com 으로 하드코딩 — 환경 무시

파일 · src/auth/useAuth.ts:68

로컬 dev / staging / preview 어느 환경에서도 무조건 production 웹 으로 돌아감. staging Supabase 프로젝트로 테스트 불가능. VITE_AUTH_REDIRECT_BASE 같은 환경변수로 분기되어야 함.

영향: Mac / Windows 빌드 시 환경별 분리 불가
HIGHHIGH-4

Supabase URL/anon key 가 하드코딩 — 에러 메시지는 .env.local 안내

파일 · src/auth/supabase.ts:3-4

코드는 URL/key 를 하드코딩하면서, 미설정 에러 메시지는 ".env.local 의 VITE_SUPABASE_URL / VITE_SUPABASE_ANON_KEY 를 확인하세요" 라고 안내. 모순. anon key 는 공개되어도 안전하지만 URL 까지 박혀 있어 env 분리가 원천 차단됨.

영향: 전 클라이언트 / 빌드/배포 위생 문제

MEDIUMMED-4

웹 PKCE 가 SDK 자동 + 수동 exchange 이중 시도

파일 · src/auth/supabase.ts:24 + src/web/Marketing.tsx:803-813

detectSessionInUrl: true 가 자동으로 code → session exchange. 그 후 AuthCallback 이 !data.session 일 때 또 한 번 exchangeCodeForSession(code) 시도. 보통은 first call 이 성공해 두 번째는 안 가지만, 순서 race 가 발생하면 두 번째가 "invalid grant" 로 실패하며 콘솔 오류. defensive 가드는 있으나 redundant 한 경로.

MEDIUMMED-5

ProfileView 의 loginLoading 이 OAuth 진행 중 풀림

파일 · src/views/ProfileView.tsx:683-741

await signInWithGoogle() 가 Tauri 분기에선 팝업 spawn + interval 등록 후 즉시 resolve. finally 의 setLoginLoading(false) 가 곧장 실행 → 사용자 입장에서 진행 표시가 사라지지만 로그인은 안 끝남. UX 혼란.

MEDIUMMED-6

macOS Dock Reopen 만 처리 — Windows 트레이 reopen 미보완

파일 · src-tauri/src/lib.rs:368-380 · src-tauri/src/tray.rs:52-65

Tray.rs 에서 Windows 트레이 클릭으로 main:reopened 를 emit 하긴 함 (tray.rs:64) — 매뉴얼하게 챙긴 것은 OK. 다만 lib.rs 주석은 macOS Reopen 만 다룸 → Windows 의 동등 경로가 tray 에 의존하는데 그 사실이 문서/주석에 명시 안 됨. 로그인 후 window reload 시점 등의 edge case 검증 필요.

MEDIUMMED-7

alwaysOnTop 팝업이 Windows 에서 공격적

파일 · src/auth/useAuth.ts:95

Mac 은 대체로 무난하나, Windows 에서 alwaysOnTop 팝업은 작업 가림 + 알림 위 표시 등 사용자 거슬림. 게다가 HIGH-1 으로 안 닫히면 더 심각.

8. 권장 수정 로드맵

Phase 1 — Tauri 로그인 부활 (Critical 해소)

  1. tauri-plugin-deep-link 도입. Cargo.toml 에 tauri-plugin-deep-link = "2" 추가, tauri.conf.json 의 plugins.deepLink.desktop.schemes: ["barosit"] 등록. macOS 의 Info.plist 자동 생성 처리.
  2. Supabase 대시보드 + Google/Kakao 콘솔에 추가 redirect URL 등록: barosit://auth-callback.
  3. useAuth.ts 의 Tauri 분기 재작성:
    • redirectTo 를 barosit://auth-callback 으로 변경.
    • 팝업 대신 shell.open(data.url) 로 사용자 기본 브라우저에서 OAuth 진행 (Tauri 표준 패턴).
    • Rust 측 deep-link handler 가 callback URL 의 토큰을 main webview 로 IPC emit.
    • main 의 supabase client 가 supabase.auth.setSession({ access_token, refresh_token }) 으로 세션 주입.
  4. 대안: 외부 브라우저 사용을 원치 않으면 Rust 측 loopback HTTP 서버 패턴 (port=0 으로 띄우고 그 port 를 redirectTo 에 동적 주입).

Phase 2 — 위생 (High 해소)

  • HIGH-3: VITE_AUTH_REDIRECT_BASE env 도입, Tauri 빌드 시에도 정상 분기.
  • HIGH-4: Supabase URL 도 import.meta.env 에서 읽도록 통일. 에러 메시지와 동작 일치시킴.
  • HIGH-1, HIGH-2: deep-link 도입 후 자연 해소되거나 dead 코드로 정리.

Phase 3 — UX (Medium 해소)

  • MED-3: WebviewWindow close event 구독해서 진행 중 OAuth 취소 시 UI 복원.
  • MED-5: signInWithGoogle() 가 실제 세션 확립까지 await 하도록 변경 (Promise 가 polling resolve 와 연결).
  • MED-2: polling 핸들을 effect/closure 로 scoping.
  • MED-7: alwaysOnTop 제거 또는 사용자 설정.
