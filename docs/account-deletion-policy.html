1. 배경 · 원칙

현재 BaroSit은 셀프서비스 회원탈퇴 기능이 없다. 프로필에 "계정 삭제·데이터 내려받기는 문의로 요청 주시면 처리"라는 안내만 있고(profile.json: noPasswordPre), 실제 삭제는 관리자가 수동 처리한다. 한국 B2C + 소셜 로그인(Google·Apple) 도입을 감안하면 셀프 탈퇴는 표준이며 곧 갖춰야 한다.

핵심 원칙

  • 즉시 완전삭제(hard delete) 금지. 실수·변심 복구, 무료체험 악용(탈퇴→재가입 반복) 방지, 법적 보존 의무 때문.
  • 유예기간(soft delete) 후 파기. 탈퇴 신청 → 접근 차단 + 유예 → 예약 작업으로 파기.
  • "전부 삭제" 아님 — 데이터 분류. 개인정보·자세 데이터는 파기, 결제·거래 기록은 법정기간 보존(분리·잠금).
  • 탈퇴를 가입만큼 쉽게. 다크패턴(숨김·고의적 난이도) 회피 — 규제 추세.

2. ⚠️ 법적 보존 의무 (즉시 삭제 불가 데이터)

개인정보보호법은 "목적 달성 시 지체없이 파기"지만, 타 법령의 보존의무가 우선한다. BaroSit은 유료 결제를 하므로 탈퇴해도 아래는 법정기간 보존해야 한다.

결론: 탈퇴 시 결제·거래 기록은 파기하면 위법. 해당 행(billing_history, user_subscriptions 등)은 법정기간까지 보존하되 가능한 한 user_id를 익명/가명화하고 다른 개인정보와 분리한다.

4. 흐름 설계 (soft delete + 유예 + 예약 파기)

1
탈퇴 신청 (ProfileView "회원탈퇴" → 강한 확인 + 비가역 경고 + 재인증 권장)
→ 계정 pending_deletion + deletion_scheduled_at 기록, 로그인/동기화/Pro 게이팅 차단, 활성 구독 자동갱신 해지.
2
유예기간 (권장 30일) — 안내 메일 발송("N일 후 영구삭제, 그 전 재로그인 시 복구 가능").
→ 유예 중 재로그인 시 탈퇴 취소 가능(pending_deletion 해제).
3
유예 경과 → 예약 파기 (pg_cron 일배치)
→ 삭제 데이터 하드 삭제 + auth 유저 삭제(service_role) + Toss 빌링키 폐기. 보존 데이터는 익명화 후 잔존.
인프라: 이미 pg_cron(charge-renewals 정기청구) 사용 중 → 동일 패턴으로 예약 파기 작업 추가 용이.

5. 구현 계획

  • Edge Function delete-account (service_role): 탈퇴 신청 처리(soft) + 본인확인. 별도 purge 배치(또는 pg_cron SQL 함수)가 유예 경과분을 실제 파기·익명화·빌링키 폐기.
  • DB: profiles에 deletion_requested_at/deletion_scheduled_at 컬럼 + 차단 RLS. user_id FK 테이블 cascade/정리 정책.
  • 프론트(ProfileView): "회원탈퇴" 버튼 + 확인 모달(데이터 영구삭제·구독·법정보존 안내) + 유예 중 "탈퇴 취소" UI. i18n ko/en/ja.
  • 결제 연계: 활성 구독자 탈퇴 시 환불/해지 정책 적용(기존 payment-cancel·subscription-manage 연계).
  • 법무 문서: privacy.md·terms에 회원탈퇴·보존기간 조항 반영.
