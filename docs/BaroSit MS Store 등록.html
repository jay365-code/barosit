실행 방식 — Claude Code가 브라우저를 직접 조작해 수행한다

이 문서는 사람에게 시키는 안내서가 아니다. Claude Code가 claude-in-chrome MCP로 Partner Center를 직접 조작해 실행하는 절차다. 페이지 이동·클릭·모든 텍스트 입력(설명·기능·키워드)·마켓 확인·인증 취소·저장·최종 제출 — 전부 에이전트가 직접 한다. 다른 세션의 에이전트도 "가이드만" 하지 말고 실제로 브라우저를 몰아서 작업할 것.

1. 사전 검증 — 재거부 방지 🔍 재확인 에이전트

패키지 만들기 전에 이 두 가지가 새 빌드에 들어있고 배포됐는지 확인한다. 안 되어 있으면 등록해도 재거부된다.

1-1. loopback 로그인 폴백 (코드)

  • Rust: src-tauri/src/lib.rs 에 start_auth_loopback/stop_auth_loopback 정의 + generate_handler![](invoke_handler)에 등록.
  • 프론트: src/auth/useAuth.ts 가 invoke("start_auth_loopback") → 브리지 URL에 ?port= 부착해 redirectTo 로 전달.
  • 브리지 페이지: public/desktop-auth-redirect.html 가 barosit:// 먼저 시도 후 실패 시 http://127.0.0.1:<port>/auth-callback 로 최상위 이동(RFC 8252).
  • 전부 커밋됨 = 그 태그로 빌드된 MSIX에 포함됨. git diff HEAD -- <파일> 로 미커밋 없음 확인.

1-2. 브리지 페이지 라이브 배포 확인

curl -sS -L "https://barosit.com/desktop-auth-redirect.html" -o /tmp/b.html   # (샌드박스면 dangerouslyDisableSandbox)
grep -n "127.0.0.1\|searchParams.get('port')\|location.replace" /tmp/b.html
diff -q public/desktop-auth-redirect.html /tmp/b.html   # 완전 일치면 최신 배포됨

Cloudflare가 .html → 클린URL로 308 리다이렉트(쿼리 보존)하는 것은 정상.

2. MSIX 빌드·획득 🔁 매 버전 에이전트

Windows 머신·GUI 불필요. GitHub Actions가 굽는다.

  1. 버전 태그(v*) 푸시하면 "Windows MSIX" 워크플로가 자동 실행(또는 gh workflow run "Windows MSIX" 수동).
  2. 완료 후 아티팩트 다운로드: 
    gh run list --workflow="Windows MSIX" -L 3         # run id 확인
gh run download <run_id> --name barosit-msix --dir ./windows/dist
ls windows/dist/*.msix                              # → BaroSit_<x.y.z.0>_x64.msix
버전 규칙 MSIX Identity 버전은 4파트, 마지막(revision)=0 (예 0.9.6.0). scripts/build-msix.ps1이 tauri.conf.json의 x.y.z를 x.y.z.0으로 변환. 현재 스토어 게시본보다 높아야 업데이트로 인식된다.
로컬 경로 주의 windows/dist·windows/layout은 gitignore. 아티팩트는 기본 90일 보관.

3. 제출 열기 (Partner Center) 🔁 매 버전 에이전트(브라우저)

Partner Center → Apps and games → BaroSit → 개요.

함정 이미 진행 중인 제출(초안/인증 중)이 있으면 "새 제출 만들기" 버튼이 없다. 한 번에 하나의 활성 제출만 허용되기 때문. 이번(0.9.6)에도 0.3.3.0이 인증 중이라 버튼이 안 보였다.
  • 인증 중인 구버전이 있는데 그걸 버리고 새 버전으로 갈 경우: 인증 상태 카드의 "인증 취소" → "예" → 제출이 초안으로 바뀜(삭제 아님) → 그 초안을 편집.
  • 진행 중 제출을 살릴 경우: 통과·게시될 때까지 기다렸다가 그 뒤 새 제출 생성.
  • 초안이 된 제출은 지난 게시본 값을 복사해 옴 → 대부분 섹션은 그대로 두고 패키지·목록만 손대면 된다.

4. 패키지 교체 🔁 매 버전 사람 직접

!

파일 업로드는 에이전트가 못 한다. 브라우저 자동화의 file_upload 도구는 (a) 세션에 첨부된 파일만 받고 (b) 10MB 한도인데, MSIX는 ~38MB. 네이티브 파일 선택창도 자동화 불가. → 담당자가 직접 올려야 한다.

  1. 제출 초안 → 패키지(Packages) 섹션 진입.
  2. 드롭영역 "browse your files" 또는 드래그로 windows/dist/BaroSit_<버전>_x64.msix 업로드(미서명 OK, MS가 재서명).
  3. 새 버전이 더 높으면 구 패키지는 "저장 시 자동 제거" 표시(취소선) → 수동 Remove 불필요.
  4. Save. 개요에서 패키지가 Validated · 업데이트됨으로 표시되면 성공.

5. 스토어 목록 — 영어·한국어·일본어 🔁 새기능만 📌 나머지 고정 에이전트(텍스트)

일반 버전 범프에선 대부분 건드릴 필요 없음. 새 제출 초안은 지난 게시본을 복사해 오므로 목록 텍스트·스크린샷은 자동으로 이어진다. 버전마다 실제로 바꾸는 값은 「이 버전의 새로운 기능」(릴리스 노트)뿐이고, 스크린샷은 UI가 바뀐 경우에만 교체한다. 아래 5-1~5-4 전체 작성은 일회성 정비(이번 0.9.6에서 한국어 목록이 비어 있고 일본어가 없어 전면 작성)에 해당 — 한 번 제대로 해두면 다음 버전부터는 생략.
이 버전의 새로운 기능✅ 매 버전 갱신 (3언어) — 이번 릴리스 변경점
설명 · 제품 기능 · 간단한 설명 · 키워드➖ 유지 (앱 컨셉/기능 변경 시에만 수정)
스크린샷➖ UI 변경 시에만 교체

각 언어별 입력 항목: 설명 · 이 버전의 새로운 기능 · 제품 기능(불릿, 최대 20) · 간단한 설명(≤약 270자) · 키워드(최대 7) · 스크린샷. 텍스트는 에이전트가 타이핑, 스크린샷은 사람이(§6). 아래는 처음부터/전면 작성이 필요할 때의 절차.

5-1. 카피 소스

앱 i18n 마케팅 카피를 재사용하면 톤·번역 품질이 앱과 일치한다: src/i18n/locales/{en,ko,ja}/marketing.json (landing/privacy/features 등), 7자세 라벨은 src/i18n/locales/*/posture.json.

5-2. 카피 규칙 (사용자 확정)

  • 한국어에 외래어 금지: "넛지", "백오프" 쓰지 말 것 → "알림 / 리마인더"로. (영어권엔 "nudge" OK)
  • 모순 회피: "나쁜 자세를 지적 안 한다"는 앱 동작과 모순. → "나쁜 자세를 짚어주긴 하지만 다그치려는 게 아니라, 한 자세가 오래 이어질 때 가볍게 알려 움직이도록 돕는다" 프레이밍. (en/ja도 동일 반영)

5-3. 언어 추가 (일본어처럼 매니페스트에 없는 언어)

매니페스트 Resources에 선언된 언어(ko-KR, en-US)는 "패키지에서 지원되는 언어"에 자동 등장. 그 외(ja-JP)는 Microsoft Store 목록 → "언어 추가/제거" → "추가 언어 관리"에서 검색·체크 후 저장하면 "추가 언어"로 편집 가능해진다.

5-4. 키워드

최대 7개, 각 ≤40자, 전체 단어수 ≤21. 입력창에 타이핑 후 Enter로 칩 생성. (AI 추천 드롭다운은 일본어에서 이상하게 나올 수 있으니 직접 입력.) 각 언어 편집 후 하단 저장.

6. 스크린샷 📌 고정 (UI 변경 시만) 사람 직접

스크린샷도 사람이 업로드 (파일 업로드 제약 동일). 단 이미지는 보통 <10MB라, 원본을 파일로 저장만 하면 에이전트가 file_upload로 올릴 여지는 있으나 — 이번엔 세션 첨부/경로 제약으로 결국 사람이 네이티브 픽커로 올림. 표준은 "사람 직접"으로 잡는다.
  • 요건: 1366×768 이상, PNG, 개당 50MB↓(스토어) / 10MB↓(에이전트 도구 시).
  • 맥 창 캡처가 1366폭 미만이면(예 1212px) 확대: sips --resampleWidth 1400 shot.png.
  • 언어별로 각각 업로드(같은 이미지 재사용 가능). 데스크톱 탭의 큰 "+" 박스에 드래그/선택.
  • 구버전 스크린샷 교체 시: 썸네일 우상단 휴지통 아이콘으로 삭제 후 새로 업로드.
  • 업로드되면 각 언어 목록의 빨간 "스크린샷 필요" 경고가 사라지고, 개요의 Store 목록이 업데이트됨이 된다(미완료 → 완료).

7. 인증 노트 · 테스트 자격증명 📌 고정 (설정 후 유지) 에이전트+사람

좌측 추가 정보 → 추가적인 테스트 정보(Additional Testing Info). 심사자만 봄(고객 비공개).

7-1. 설명(테스트 지침) — 에이전트 작성 후 "Save description"

1차 거부가 로그인이었으므로 핵심 메시지는 "핵심 기능은 로그인 없이 심사 가능":

  • ① 자세 감지는 100% 온디바이스 → 웹캠 허용만 하면 로그인 없이 전 기능 리뷰 가능.
  • ② 카메라 권한 필요(첫 실행 시 허용).
  • ③ 로그인은 선택(동기화/커뮤니티용). 이전 "브라우저 로그인 후 앱 미복귀" 이슈는 127.0.0.1 loopback 폴백(RFC 8252)으로 수정됨. 검증하려면 아래 자격증명 사용(이메일 로그인).

7-2. 자격 증명(Credentials)

비밀번호는 에이전트가 입력 금지 — 안전 규칙상 AI는 비밀번호를 어떤 입력창에도 타이핑하지 않는다. 이름(라벨)·이메일은 에이전트가 넣고, 비밀번호 값은 담당자가 직접 입력한다.
  • "새 자격 증명"으로 2개 추가: 이름 Email (ID) / 값 reviewer@barosit.com (에이전트), 이름 Password (email login) / 값 = 비번(사람 직접).
  • 추가 직후 "Unable to save" 배너가 뜨면 한 번 더 시도하면 반영되는 경우 있음(이번에 그랬음). 표에 ******로 마스킹돼 나오면 저장된 것.

7-3. 게시 옵션

제출 옵션(Options): 기본값 "인증 통과 즉시 게시" 유지. 특정 날짜 예약이 필요하면 여기서.

8. 최종 검증 · 제출 🔍 China 재확인 🔁 제출 에이전트, 제출은 사용자 승인

8-1. China 마켓 제외 확인 (필수)

  • 가격 책정 및 가용성 → 지역/국가. 라디오가 "전 세계 모든 시장"이어도, "옵션 표시"로 목록을 열어 "중국(ISBN 필수)" 체크가 해제돼 있는지 확인. (검색창에 중국)
  • 제외돼 있으면 창을 취소로 닫아(변경 없이) 상태 유지.

8-2. 제출 전 체크리스트

  • 패키지 = 새 버전(x.y.z.0), Validated·업데이트됨, 구버전 제거됨
  • en·ko·ja 목록 모두 업데이트됨(설명·기능·간단설명·키워드·스크린샷)
  • loopback 코드+브리지 배포 검증 완료(§1)
  • China 마켓 제외 유지
  • 인증 노트 + 테스트 자격증명(Email/Password) 등록
  • 게시 옵션 = 인증 통과 즉시
제출 개요의 파란 "인증을 위해 제출" 클릭. 되돌리기 어려운 공개 액션이므로 사용자 승인 후 실행. 누르면 상태가 제출 → 사전 처리 중 → 인증 → 게시 중으로 진행. 인증은 수 시간~최대 3영업일, 통과 시 자동 게시. 필요 시 "인증 취소"로 되돌릴 수 있음.

참고 — 일본어가 바로 "스토어 목록만" 케이스였다. 앱엔 ja 로케일이 있지만 매니페스트 <Resources>엔 ko-KR·en-US만 선언돼 있어, ja는 "추가 언어"로 붙여 재빌드 없이 처리했다. 스페인어도 같은 패턴이면 재빌드 불필요.

  • languageid: 추가 후 목록 URL의 languageid를 기록해두면 다음에 바로 접근(현재 ko=61·en=4·ja=17).

  • 키워드 AI 추천이 일본어에서 프롬프트 유출성 텍스트로 나옴 → 무시하고 직접 입력.

China 재포함 백로그 (보류 중)

이메일 로그인 도입으로 1차 거부(소셜 로그인 미도달)는 해소되지만, 아래 리스크로 0.9.6 제출에선 제외 유지 결정. 재포함하려면 선행 필요:

  • 중국어(zh) 스토어 목록 추가(현재 없음 → 중국 노출 부적합).
  • Supabase(AWS) 중국 도달성 확인 — 이메일 로그인/동기화 경유. 불안정하면 심사 중 로그인 실패 위험.
  • 폰트 CDN 로컬화 — 랜딩/헬퍼 페이지가 cdn.jsdelivr.net(중국 차단)에서 Pretendard 로드. 앱 본체는 시스템폰트 폴백이라 치명적이진 않음.
  • 참고: MediaPipe는 로컬 번들(src/pose/detector.ts, ${BASE_URL}mediapipe)이라 자세감지는 중국 무관.

제출 단위로 인증되므로, 중국 특유 문제 하나가 전체 릴리스 거부로 번질 수 있음 → China는 별도 제출로 분리하는 편이 안전.
