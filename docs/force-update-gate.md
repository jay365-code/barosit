# 강제 업데이트 게이트 (Force-Update Gate)

구버전 앱이 "업데이트해야 계속 이용 가능"하도록 막는 장치. 긴급 장애 버전을 빠르게
사용자에게서 걷어내기 위한 것. 경보성 문구 없이 담백한 "업데이트 필요" 게이트.

3단계로 나뉘며 **1차만 구현**되어 있다. 2·3차는 아래 트리거가 왔을 때 추가한다.

| 단계 | 무엇 | 강제력 | 상태 |
|---|---|---|---|
| **1차** | 클라이언트 하드 게이트 (fail-open) | UX 레벨(클라가 스스로 막음) | ✅ 구현됨 |
| **2차** | 서버 게이트 (엣지 함수가 구버전 요청 거부) | 진짜 강제력(변조 우회 불가) | ⬜ 계획 |
| **3차** | Realtime 즉시성(킬 스위치) | 속도 | ⬜ 계획 |

---

## 핵심 안전 원칙 (모든 단계 공통)

1. **fail-open** — "구버전임을 원격 설정으로 확인했을 때만" 차단한다. 오프라인·백엔드
   장애·설정 없음·잘못된 값이면 **절대 막지 않는다**. 이유: 자세 감지는 온디바이스라
   오프라인에서 도는 게 핵심 가치이며, fail-closed 면 Supabase 한 번 다운에 전 사용자가
   잠긴다. 문제 버전은 온라인이 되어 설정을 읽는 순간 차단되므로 목적은 그대로 달성된다.
2. **탈출로** — 자동 업데이트가 권한 문제 등으로 실패해도 갇히지 않도록 **수동 다운로드
   링크**를 항상 제공한다.
3. **관리자 전용 + 신중** — `min_supported_version` 값 하나 잘못 넣으면 (온라인) 전
   사용자가 락아웃된다. 설정은 관리자만, 값 재확인 필수.
4. **버전 비교는 공용 유틸** — `src/lib/semver.ts` 를 클라·서버가 함께 쓴다(문자열
   비교 함정 회피: "0.9.9" < "0.9.10").

---

## 1차 — 클라이언트 하드 게이트 (구현됨)

### 동작
- `app_config.min_supported_version` 을 읽어 실행 버전이 그보다 낮으면 앱 전체를 덮는
  차단 화면(`UpdateRequiredGate`)을 띄운다. 다른 모든 화면(admin/profile/main)보다 우선.
- 화면: "업데이트가 필요합니다" + [지금 업데이트](자동 다운로드·설치·재시작) + 수동
  다운로드 링크. 진행률 바. 경보성 문구 없음.
- 부팅 시 1회 + 세션 중 30분마다 재평가(부팅 후 설정이 켜져도 결국 걸리도록).
- 웹은 항상 최신 번들이라 자연히 no-op(실행 버전 < min 이 거의 성립 안 함).

### 구성 파일
- `src/lib/semver.ts` — 버전 파싱·비교 (`isBelowMinVersion`, fail-open). +테스트.
- `src/updateGate.ts` — `useUpdateGate()` 훅. app_config 읽기 + 비교. **확장 단일 지점.**
- `src/components/UpdateRequiredGate.tsx` — 차단 화면.
- `src/App.tsx` — `updateGate.blocked` 면 최우선 early-return.
- i18n: `app.updateRequired.*` (ko/en/ja).

### 운영법 — 긴급 차단 켜기/끄기
Supabase `app_config` 테이블(키-값)에 행 하나:

```sql
-- 0.9.10 미만을 전부 차단 (0.9.9 장애 → 0.9.10 배포 후)
insert into app_config (key, value) values ('min_supported_version', '0.9.10')
  on conflict (key) do update set value = excluded.value;

-- 해제 (평상시로 복귀)
delete from app_config where key = 'min_supported_version';
-- 또는 value 를 '0.0.0' 으로 낮춰도 됨.
```

- 값이 없으면(기본) 게이트 비활성 = 아무도 안 막힘.
- 설정 후 사용자는 **다음 재평가(최대 30분) 또는 앱 재시작 시** 차단된다. 즉시성이
  필요하면 3차(Realtime) 필요.
- **반드시 그 버전이 GitHub Release 로 이미 나와 있을 때** 값을 올릴 것 — 안 그러면
  사용자가 [지금 업데이트]를 눌러도 받을 게 없다(수동 링크로는 빠져나감).

## 2차 — 서버 게이트 (계획)

### 언제 추가하나 (트리거)
- 클라 게이트를 변조로 우회하는 사용자가 실제 관측됨, 또는
- **결제·데이터 오염**처럼 "구버전이 백엔드와 통신하는 것 자체가 위험한" 장애 발생
  (가장 현실적 트리거), 또는
- 규모가 커져 구버전 클라가 서버에 주는 리스크가 무시 못 할 수준.

### 설계
- 클라가 모든 요청에 `X-Client-Version` 헤더(또는 함수 인자)로 앱 버전 전달.
- 민감한 엣지 함수(결제부터 → 데이터쓰기 → 커뮤니티/AI)가 진입부에서 버전 검사:
  구버전이면 **HTTP 426 Upgrade Required**(또는 `{ error: "upgrade_required" }`) 반환.
  기준 버전은 `app_config.min_supported_version` 재사용, 비교는 `lib/semver` 공유.
- 클라는 426/`upgrade_required` 응답을 받으면 `useUpdateGate` 를 강제로 켠다
  (1차 훅에 "서버가 막았다" 진입점만 추가하면 됨 — 게이트 UI 재사용).
- **범위: 네트워크 기능만.** 온디바이스 감지 코어는 건드리지 않음(오프라인 우선 유지).
- 주의: 정상 요청이 깨지지 않도록 롤아웃 신중히. 헤더 누락 시 fail-open.

## 3차 — Realtime 즉시성 (계획)

### 언제 추가하나 (트리거)
- 실제 긴급 장애에서 1차의 "다음 재평가까지 최대 30분" 지연이 너무 느리다고 판단될 때.
  (한 번이라도 "지금 당장 전원 차단해야 하는데 기다려야 하네"를 겪으면.)

### 설계
- Supabase Realtime 으로 `app_config`(key=`min_supported_version`) 행을 구독.
- 값이 바뀌면 `useUpdateGate` 의 `evaluate()` 를 즉시 재실행 → 켜져 있는 앱들이 수 초
  내 차단 화면으로 전환. 30분 폴링은 백업으로 유지.
- 구현은 `useUpdateGate` 안에 구독 하나 추가로 끝남(훅이 단일 관리 지점이라 저비용).
